# 仪表盘状态总览重设计 — 设计文档

**日期：** 2026-04-21  
**主题：** 将首页仪表盘从简化入口页升级为“全局汇报优先”的状态总览页  
**状态：** 待实现  
**依赖：** 现有 Vue 3 前端、Pinia store、现有小说状态与内容查询 API

---

## 1. 目标

当前仪表盘页过于简化，只展示少量摘要信息和操作入口，无法回答以下关键问题：

- 这本书现在整体推进到哪一步了？
- 当前卷和当前章的工作状态如何？
- 知识数据是否在持续积累？
- 现在最适合执行的下一步动作是什么？
- 如果流程异常，异常大致发生在哪一层？

本次重设计的目标是将首页改造成一个**总览页**，优先满足：

1. **汇报全局**：用户进入首页后，先快速理解整本书当前推进情况。
2. **指导下一步**：在不进入深层页面的前提下，给出当前最合适的动作入口。
3. **暴露问题**：用轻量方式提示阻断项、缺失项与最近失败事件。

---

## 2. 设计原则

### 2.1 信息优先级

首页信息优先级固定为：

1. 全局状态汇报
2. 下一步动作建议
3. 异常与风险提醒

因此首页不能演化为“按钮墙”或“实时日志墙”，而应以状态叙事为主。

### 2.2 首页边界

首页默认只展示：

- 小说级总览
- 当前卷级摘要
- 当前卷章节进度摘要

首页**不展开所有章节明细表**，也不承载深度编辑能力。所有明细查询通过二级页面承接。

### 2.3 数据状态展示方式

“数据状态”采用**混合型摘要**：

- 先展示完成度与覆盖情况
- 再展示最近新增或最近变化

这样既能回答“有没有在积累”，也能回答“最近具体更新了什么”。

### 2.4 运行状态刷新策略

首页采用**轻实时模式**：

- 关键流程状态与最近动作结果自动刷新
- 内容型数据低频刷新，或在动作完成后刷新
- 不在首页持续滚动完整日志流

---

## 3. 页面信息架构

首页采用 `A 总览汇报型` 作为主骨架，吸收 `B 指挥台型` 的动作区能力，形成 4 层结构。

### 3.1 顶部总览带

职责：只回答“现在整体走到哪里了”。

展示内容：

- 小说标题
- 当前阶段
- 当前卷 / 当前章
- 累计总字数
- 已归档章节数

这一层保持高度压缩，不放复杂解释。

### 3.2 核心状态卡区

首页中部第一屏展示 4 张摘要卡：

1. `流程进度`
2. `章节进度`
3. `数据状态`
4. `运行状态`

每张卡统一遵循两个输出：

- 一个整体结果
- 一个最重要的提示

例如：

- `流程进度`：当前处于“卷规划”，上一阶段“脑暴”已完成
- `数据状态`：伏笔 18 条，其中 2 条待回收
- `运行状态`：最近一次推进成功，3 分钟前

### 3.3 当前卷工作区

当前卷工作区分为左右两栏。

左栏：`当前卷章节进度摘要`

- 当前卷章节状态分布
- 当前章高亮
- 当前卷累计字数
- 当前章评分或目标完成度摘要

右栏：`下一步建议动作`

- 1 个主动作
- 1 到 2 个次动作
- 每个动作附带简短原因说明

这一块是首页吸收 `B 指挥台型` 的唯一重点区域，避免整页变成以操作为中心。

### 3.4 底部洞察区

底部区域承担“继续深入”的作用，分为 3 个部分：

- 最近更新
- 风险提醒
- 快速跳转

其中：

- 最近更新偏内容变化
- 风险提醒偏阻断与缺失
- 快速跳转负责把用户导向章节、实体、时间线、伏笔、日志等明细页

## 4. 模块级内容定义

### 4.1 流程进度卡

展示字段：

- 当前阶段
- 上一阶段结果
- 下一阶段预期动作
- 当前阶段是否存在阻断提示

示例文案：

- “已完成分卷，等待生成上下文”
- “当前阶段为归档中，上一轮编辑已完成”

### 4.2 章节进度卡

展示字段：

- 当前卷章节总数
- 已完成数量
- 处理中数量
- 待开始数量
- 当前章标题
- 当前卷累计字数

表现形式建议为：

- 压缩进度条
- 状态分布标签
- 当前章名称强调

不在卡内展开完整章节表。

### 4.3 数据状态卡

数据状态卡采用“完成度 + 最近更新”双层结构。

完成度摘要展示：

- 实体总数
- 时间线条目数
- 伏笔总数
- 待处理资料数

最近更新展示：

- 最近新增的实体 / 伏笔 / 时间线 / 资料处理结果中的 2 到 4 条

设计要求：

- 若某类数据为空，应明确表达“暂无数据”
- 不把“无数据”误报为“加载失败”

### 4.4 运行状态卡

展示字段：

- 最近一次动作
- 动作结果（成功 / 失败 / 处理中）
- 更新时间
- 自动刷新状态

若最近有失败，卡内增加一条简短失败摘要，例如：

- “实体提取失败：响应解析异常”

第一版数据来源约束：

- 优先基于现有 `loadingActions`、当前阶段变化和 `/logs/stream` 中最近日志派生
- 若首页无法稳定得到“最近一次动作结果”，则允许先展示最近日志摘要，不强行新增后端接口

完整日志仍然保留在日志页。

### 4.5 当前卷章节进度摘要

当前卷工作区左栏建议组合以下元素：

- 当前卷章节状态分布
- 当前章卡片
- 小型字数图或评分图

优先回答：

- 当前卷推进到了哪里
- 当前章是否需要关注

### 4.6 下一步建议动作

动作区只保留少量高信号操作：

- 1 个主动作
- 1 到 2 个备选动作

动作区必须附带简短理由，不允许只展示按钮名称。例如：

- “建议先生成上下文，因为当前已完成卷规划，且当前章尚未准备上下文”

现有 `ActionPipeline` 不应原样平铺到首页，而应重新包装为“推荐动作”模块。

### 4.7 最近更新 / 风险提醒 / 快速跳转

最近更新：

- 展示最近发生的内容变化，而不是系统事件流水

风险提醒：

- 只展示会影响决策的高优先级异常
- 普通提醒不抢首页主视觉

快速跳转：

- 章节列表
- 实体百科
- 时间线
- 伏笔
- 实时日志

## 5. 交互与刷新策略

### 5.1 自动刷新范围

自动刷新的内容：

- 当前阶段
- 当前卷 / 当前章
- 最近动作结果
- 运行状态卡

低频刷新或动作后刷新的内容：

- 实体
- 时间线
- 伏笔
- 资料处理状态

### 5.2 卡片交互

每张摘要卡只提供一个主入口：

- 章节进度卡 → 章节列表
- 数据状态卡 → 对应内容页
- 运行状态卡 → 实时日志
- 风险提醒 → 对应异常来源页

首页不承担复杂操作流，只负责发现、说明与分发。

### 5.3 异常展示层级

异常提示分 3 级：

1. **阻断型异常**
   - 例如当前阶段失败、关键数据缺失、当前章不存在
   - 提升显示优先级

2. **重要提醒**
   - 例如待处理资料堆积、最近动作失败
   - 进入风险提醒区或卡片提示

3. **普通状态信息**
   - 例如最近刷新时间
   - 保持轻量展示

### 5.4 日志策略

首页只展示最近几条运行摘要，不内嵌完整控制台。

完整日志流继续由日志页承载，避免首页进入“监控台”模式。

## 6. 现有结构复用与实现边界

### 6.1 可复用能力

现有前端已具备以下基础：

- `novel store` 中已有 `novelState`、`archiveStats`、`chapters`、`currentChapter`
- 已有 `entities`、`timelines`、`foreshadowings`、`pendingDocs` 等内容数据
- 已有 `ActionPipeline`、`ScoreRadar`、`ChapterProgressGantt`、`LogConsole` 等基础组件
- 已有章节、实体、时间线、伏笔、日志等二级页面路由

### 6.2 首页拆分建议

不建议继续把逻辑堆在单一 `Dashboard.vue` 中，建议拆分为多个首页组件：

- `DashboardHero`
- `DashboardStatusCards`
- `DashboardVolumeSummary`
- `DashboardNextActions`
- `DashboardInsights`

这样可以降低首页复杂度，也便于后续继续增加状态块。

### 6.3 第一版实现边界

第一版仅消费现有 API 和 store 能直接或经前端轻度聚合得到的数据。

优先策略：

- 能前端聚合的先前端聚合
- 不为首页一开始新增大量专用 API

仅在以下情况下考虑新增首页摘要接口：

- 现有 API 请求过多，导致首页加载明显冗余
- 首页必须展示的摘要无法通过现有 store 稳定得出

特别是运行状态卡，第一版优先复用现有实时日志流和动作状态，不单独为其设计新协议。

### 6.4 明确不包含

本次设计明确不包含：

- 跨小说聚合面板
- 所有章节全量状态总表
- 首页深度编辑
- 首页完整实时日志控制台
- 为首页而做的大规模后端重构

## 7. 失败、空状态与降级策略

### 7.1 卡片级降级

若某一块数据加载失败，不允许整页一起失败，应按卡片降级。

例如：

- `数据状态卡` 加载失败时，仅该卡显示失败提示与重试入口
- 其他模块继续可用

### 7.2 空状态区分

必须严格区分以下两类情况：

- 暂无数据
- 加载失败

示例：

- “暂无时间线数据” 表示当前没有内容
- “时间线加载失败” 表示请求或解析异常

### 7.3 未选择小说

当用户未选择小说时，首页显示明确空状态引导，不渲染半残缺仪表盘。

## 8. 测试与验收标准

### 8.1 测试重点

前端实现后至少覆盖以下场景：

1. 完整数据下的首页总览展示
2. 部分模块为空时的降级展示
3. 当前阶段变化后推荐动作随之变化
4. 自动刷新后关键摘要可更新

若当前项目暂未完善前端组件测试基建，则至少应覆盖：

- store 聚合逻辑
- 推荐动作计算逻辑
- 关键显示条件分支

### 8.2 首页验收问题

用户打开首页后，应能在 5 秒内回答以下问题：

- 这本书现在推进到哪一步了？
- 当前卷和当前章是什么状态？
- 数据有没有在持续积累？
- 我下一步最该点什么？
- 如果有问题，问题大概在哪？

若上述问题仍无法快速回答，则说明首页仍然过于稀薄、分散或信息层级错误。

---

## 9. 实施建议

建议按以下顺序实施：

1. 重构首页结构与组件边界
2. 先接入现有 store 数据，完成静态摘要布局
3. 增加推荐动作与风险提醒逻辑
4. 增加轻实时刷新
5. 根据实际数据不足情况评估是否补充首页摘要接口

该顺序可以先解决“页面太简单、状态不够清楚”的核心问题，再决定是否需要后端补强。
