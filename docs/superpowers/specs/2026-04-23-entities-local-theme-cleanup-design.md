# 实体百科局部主题整理设计

## 背景

当前实体百科页面整体已经是深色工作台风格，但局部区域仍然混用了多套颜色语义，导致视觉冲突集中出现在以下位置：

- 左侧目录中的实体数量标签与待确认标签
- 右侧详情中的“人工覆盖”表单区
- 实体信息描述表格
- “分类依据”信息卡

这些区域的问题不是布局错误，而是颜色语言不统一：

- 工作台背景和卡片使用深色玻璃风格
- 表单控件和部分按钮仍保留 Element Plus 的默认亮色语义
- 描述表格和提示卡保留接近浅色后台的底色分层

结果是页面主基调已经确定，但局部控件像来自不同产品，造成“颜色不协调”的直接观感。

## 目标

本次改动只解决实体百科页面中最明显的局部配色冲突，不扩展为全站换肤系统。

首轮目标：

- 统一截图中四类重点区域的颜色语言
- 保持深色工作台基调不变
- 用语义化变量组织颜色，为后续扩成换肤能力预留接口
- 尽量限制改动范围，避免影响其它页面

本次不做：

- 不实现全站主题切换
- 不实现浅色主题
- 不重构全局 `App.vue` 外壳
- 不调整与截图无关的工作台页面

## 方案比较

### 方案 A：仅做局部样式修补

直接在几个组件里写死更顺眼的颜色覆盖。

优点：

- 见效最快
- 代码改动最少

缺点：

- 后续想扩展成换肤时需要返工
- 颜色定义会散落在多个组件里

### 方案 B：实体百科局部主题层

在实体百科范围内新增一层语义化主题变量，统一驱动截图中的几个高冲突区域，但不把能力扩展到全站。

优点：

- 当前问题能直接解决
- 范围可控
- 后续如需做浅色皮肤或推广到其它工作台页面，可以沿用变量命名

缺点：

- 相比纯补丁，需要先整理少量 token 边界

### 方案 C：直接做工作台换肤系统

建立页面级主题状态、双皮肤和跨页面接入。

优点：

- 架构最完整

缺点：

- 超出当前用户明确边界
- 会把任务扩展到其它页面和全局容器

## 决策

采用方案 B：只为实体百科建立局部主题层，先解决用户框出的区域，同时保留未来扩展成换肤能力的命名和结构。

## 设计

### 1. 主题边界

主题变量只在实体百科相关组件内生效，不改全局 `:root` 主题策略。

建议在实体百科页面根节点下增加局部作用域类，例如：

- `entities-theme`

所有本次新增颜色变量都挂在这个作用域下，避免污染全站。

### 2. 语义变量

不直接在组件里写死颜色值，而是在局部主题层定义语义变量。

建议最小变量集合：

- `--entities-panel-bg`
- `--entities-panel-bg-soft`
- `--entities-panel-bg-muted`
- `--entities-panel-border`
- `--entities-panel-border-strong`
- `--entities-text`
- `--entities-text-muted`
- `--entities-text-soft`
- `--entities-accent`
- `--entities-accent-soft`
- `--entities-warning`
- `--entities-warning-soft`
- `--entities-danger`
- `--entities-chip-bg`
- `--entities-chip-border`
- `--entities-grid-label-bg`
- `--entities-grid-content-bg`
- `--entities-info-bg`

这些变量只覆盖当前问题所需的层级、文本、强调、警示和表格语义。

### 3. 视觉方向

#### 3.1 页面总基调

继续使用深色工作台，不改页面背景方向。

重点调整原则：

- 降低纯白区域面积
- 降低默认蓝按钮的突兀程度
- 让信息面板形成稳定的三层深色结构

#### 3.2 目录标签

目录中普通数量标签从“亮白小胶囊”调整为：

- 深色底上的低对比中性色标签
- 轻边框
- 文本比当前略暗，避免喧宾夺主

`待确认` 标签保留暖色提示，但改成低饱和暖底 + 暖色文字，不再使用高亮浅底。

#### 3.3 人工覆盖区

人工覆盖区内的输入框和选择框不再使用接近纯白的底色，改为：

- 面板内次级深色表面
- 与外层卡片有轻微层级差
- 通过边框和焦点态区分可交互性

按钮语义统一为：

- `保存覆盖`：唯一主按钮，使用收敛后的青蓝强调色
- `清除覆盖`：中性次按钮
- `重新判断`：中性次按钮，不与主按钮争夺注意力

#### 3.4 详情表格

当前表格标签列偏灰、内容列偏白，浅深冲突明显。调整为：

- 标签列使用略深的网格背景
- 内容列使用同体系的深色内容背景
- 边框改为低对比描边
- 文本改为高可读浅色

这样表格看起来属于同一个深色工作台，而不是浅色描述表格强行嵌入。

#### 3.5 分类依据卡

“分类依据”卡从大面积亮白信息框改成：

- 统一信息面板底色
- 保留提示属性，但不脱离当前主题
- 图标、标题、正文分层更清楚

## 组件落点

本次只处理与截图直接相关的几个组件：

- `src/novel_dev/web/src/views/Entities.vue`
- `src/novel_dev/web/src/components/entities/EntityTree.vue`
- `src/novel_dev/web/src/components/entities/EntityDetailPanel.vue`
- `src/novel_dev/web/src/components/entities/EntityGroupTable.vue`
- 如有必要，补充 `src/novel_dev/web/src/style.css` 中的局部主题变量定义

改动原则：

- 变量定义集中
- 组件内只消费变量
- 不把实体百科的颜色规则写回全局工作台公共规则

## 实现步骤

1. 在实体百科页面根节点建立局部主题作用域类。
2. 在样式层定义实体百科专用语义变量。
3. 调整左侧树节点计数标签和待确认标签的底色、边框和文字颜色。
4. 调整人工覆盖区的容器、输入控件和按钮语义。
5. 调整详情描述表格的标签列、内容列和边框分层。
6. 调整“分类依据”信息卡的背景和文本层级。
7. 检查是否有 Element Plus 默认变量导致漏改，并在局部作用域中补齐覆盖。

## 风险与处理

### 风险 1：Element Plus 默认样式回流

一些控件颜色来自组件库内部变量，单纯改外层容器可能不生效。

处理：

- 在局部作用域中用 `:deep(...)` 覆盖相关内部节点
- 优先覆盖组件变量，避免堆叠过多选择器

### 风险 2：局部主题变量与全局变量混淆

如果直接复用 `--app-*` 并在组件内叠加写死颜色，后续难以维护。

处理：

- 本次明确使用 `--entities-*` 前缀
- 局部组件只使用这一层语义变量

### 风险 3：目录区和详情区层级不一致

即使单个组件变好看，若左侧和右侧的明暗层级不一致，整体仍会散。

处理：

- 将目录标签、表单区、表格区都纳入同一套深色分层规范
- 用同一组边框和文本对比度控制它们的统一感

## 测试与验收

主要验收方式以界面观察为主，辅以现有前端测试不回归。

验收标准：

- 用户截图中标出的四类区域不再出现大面积亮白控件
- 按钮强调关系明确，只有一个主动作最突出
- 表格、信息卡、表单属于同一深色工作台体系
- 不影响实体百科原有交互与数据流

建议执行：

- 相关前端单测
- 本地打开实体百科页面，人工检查深色界面的实际观感

## 范围结论

这次是一次局部主题整理，不是全站换肤功能开发。

如果这次整理效果满意，下一步再决定是否把 `--entities-*` 语义变量抽象成工作台级主题 token，并扩展到其它页面。
