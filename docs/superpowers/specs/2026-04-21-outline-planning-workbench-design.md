# 大纲规划工作台 — 设计文档

**日期：** 2026-04-21  
**主题：** 将现有“卷规划”页面升级为“`大纲规划`”工作台，支持总纲与卷纲的左右布局浏览，以及页面内对话式优化  
**状态：** 待实现  
**依赖：** 现有 `VolumePlan.vue`、`BrainstormAgent`、`VolumePlannerAgent`、`/synopsis` 与 `/volume_plan` 接口

---

## 1. 目标

当前页面 `VolumePlan.vue` 只是在同一页顺序展示总纲和当前卷规划，缺少三类能力：

- 用户无法在一个统一入口里查看“总纲 + 各卷卷纲”的全貌。
- 用户只能手动编辑 `SynopsisData` JSON，不能直接用自然语言提出修改意见。
- 页面缺少以“大纲项”为中心的上下文，无法围绕某个总纲或某一卷持续迭代。

本次升级目标是把该页面改造成一个面向大纲创作的工作台：

- 页面名称从 `卷规划` 改成 `大纲规划`。
- 页面主体改为左右结构。
- 左侧展示统一的大纲列表：`总纲` + `每一卷卷纲`。
- 右侧同时承载“当前版本详情”和“对话式修改”。
- 用户可以针对总纲或任意一卷卷纲输入修改意见，系统调用对应 agent 优化结果。
- 每个大纲项拥有独立会话和独立上下文，避免不同卷之间的对话污染。

---

## 2. 范围

### 2.1 明确包含

- 页面标题、导航文案、主视图从 `卷规划` 统一改为 `大纲规划`
- 左侧大纲列表
- 右侧详情区
- 右侧会话区
- 针对 `总纲` 的对话式优化
- 针对 `单卷卷纲` 的对话式优化
- 会话级上下文压缩与摘要
- 前后端支持按大纲项读取详情、读取历史、提交优化请求

### 2.2 明确不包含

- 通用聊天中心
- 流式 token 输出
- 多人协作
- 历史版本 diff / rollback
- 页面内直接编辑完整 JSON 作为主流程
- 同时并发优化多个大纲项

---

## 3. 用户旅程

### 3.1 查看与切换

1. 用户进入 `大纲规划` 页面。
2. 左侧看到 `总纲` 和若干卷项。
3. 点击任一项后，右侧加载该项详情和该项独立会话。

### 3.2 修改总纲

1. 用户选中 `总纲`。
2. 在右下输入“修改意见”。
3. 前端提交该意见。
4. 后端调用 `BrainstormAgent`，基于总纲当前结果与会话上下文进行优化。
5. 右上更新总纲详情，右下追加一轮消息记录。

### 3.3 修改某卷卷纲

1. 用户选中某一卷。
2. 若该卷已有卷纲，右上显示当前卷纲；若没有，则显示空状态。
3. 用户输入“请强化第 3 卷宗门大战前的伏笔铺垫”之类的意见。
4. 后端调用 `VolumePlannerAgent`，返回新的卷纲结果。
5. 页面刷新该卷详情，并保留该卷自己的会话历史。

## 4. 页面信息架构

整体采用左右结构：

```text
┌──────────────────┬──────────────────────────────────────────────┐
│ 左侧：大纲列表   │ 右侧：当前大纲工作区                         │
│                  │                                              │
│ - 总纲           │ 上半区：结构化详情 / 空状态 / 优化中状态      │
│ - 第 1 卷        │                                              │
│ - 第 2 卷        │ 下半区：当前项独立会话 + 输入框              │
│ - 第 3 卷        │                                              │
└──────────────────┴──────────────────────────────────────────────┘
```

### 4.1 左侧：大纲列表区

职责：

- 作为页面内导航
- 作为轻量状态面板
- 明确当前正在编辑的是哪个大纲项

列表固定包含：

- `总纲`
- `第 N 卷`

即使卷纲尚未生成，也应显示占位项，方便用户点进去发起生成或优化。

### 4.2 右侧：详情工作区

分为上下两块：

- 上半区：当前选中大纲项的结构化结果
- 下半区：当前选中大纲项的对话交互区

结果区和对话区必须分层，避免用户每次回来先陷入聊天记录。

---

## 5. 左侧列表模型

前端不直接拼 `synopsisData + volumePlan`，而是消费统一的 `outline items`。

每个 item 至少包含：

- `outline_type`: `synopsis` | `volume`
- `outline_ref`: 总纲固定值或卷号
- `title`
- `status`: `ready` | `missing` | `generating` | `updating` | `error`
- `updated_at`
- `message_count`
- `summary_hint`
- `is_current`

状态定义：

- `ready`
  - 当前项已有可展示结果
- `missing`
  - 卷纲尚未生成
- `generating`
  - 首次为该卷创建卷纲
- `updating`
  - 当前项已有结果，但正在根据最新意见优化
- `error`
  - 最近一次优化失败

列表项 UI 展示三层信息：

- 主标题：如 `总纲`、`第 2 卷`
- 压缩摘要：如“已更新主线冲突”“尚未生成卷纲”
- 右侧状态：更新时间或状态徽标

第一版不要求真正的“未读消息系统”，但字段设计允许后续扩展。

---

## 6. 右侧详情区设计

### 6.1 总纲详情

当选中 `总纲` 时，展示：

- 标题
- 一句话梗概
- 核心冲突
- 主题
- 人物弧光
- 剧情里程碑
- 预估卷数 / 总章数 / 总字数
- 原始文本或 JSON 折叠区

### 6.2 卷纲详情

当选中 `卷纲` 时，展示：

- 卷标题
- 卷摘要
- 本卷目标
- 本卷核心冲突或推进点
- 章节数 / 预估总字数
- 章节摘要列表
- 如已有评分结果，则展示卷纲评分摘要
- 原始文本或结构化 JSON 折叠区

### 6.3 空状态

对尚未生成的卷纲，不显示空白页，而是展示：

- “当前卷尚未生成卷纲”
- 一条说明文案
- 仍然可用的对话输入区

因此详情区必须支持 3 种主状态：

- 已有内容
- 正在生成或优化
- 暂无内容

## 7. 对话交互设计

### 7.1 会话边界

`总纲` 与 `每一卷卷纲` 各自独立会话：

- 总纲有自己的对话历史
- 每一卷卷纲也各有自己的对话历史
- 切换左侧项时，右侧切换到对应会话与详情

这是本设计的核心约束。不同卷之间不得共用消息历史。

### 7.2 对话形态

前端表现为连续对话，但后端执行按“逐轮优化请求”落地：

- 用户发一条意见
- 系统执行一轮优化
- 返回新的结构化结果与一条系统回复

第一版不做 token 级流式输出。

### 7.3 消息结构

每轮至少记录：

- 用户意见
- 系统结果说明
- 可选元信息：调用的 agent、执行耗时、是否成功

系统消息不应只有“成功/失败”，而应包含压缩变更说明，例如：

- “已强化第二卷主线冲突，并提前埋入终局伏笔”
- “未能完成调整，原因是当前卷纲缺少章节骨架”

### 7.4 输入区规则

- 支持多行输入
- 执行中禁用发送按钮
- 同一会话不允许并发提交
- 切换大纲项时不强拦截未发送内容，但可给轻提示

## 8. Agent 调用策略

### 8.1 路由规则

- 选中 `总纲` 时，提交意见调用 `BrainstormAgent`
- 选中 `卷纲` 时，提交意见调用 `VolumePlannerAgent`

### 8.2 调用目标

不是重新从零生成整本书，而是围绕“当前大纲项的最新结果”做定向优化。

### 8.3 上下文组装

每次请求上下文由 4 部分组成：

1. `当前目标对象`
   - 当前是总纲还是第几卷卷纲
   - 当前对象最新结构化结果
   - 当前对象最近状态

2. `必要上游上下文`
   - 修改卷纲时带入总纲摘要
   - 如有必要，带极简前卷摘要
   - 第一版不要求带所有卷纲全文

3. `最近对话窗口`
   - 只带最近若干轮消息

4. `持久摘要`
   - 存储该会话长期修改偏好与已确认方向

### 8.4 上下文原则

- 不把整段历史聊天原文全部回灌
- 不把无关卷纲全文一并塞进 prompt
- 优先带“当前最新结果 + 会话摘要 + 最近几轮消息”

上下文压缩与 prompt 组装落在后端服务层，而不是前端。

## 9. 后端设计

### 9.1 新增对象

建议新增两类持久化对象：

#### `outline_session`

字段建议：

- `id`
- `novel_id`
- `outline_type`
- `outline_ref`
- `status`
- `conversation_summary`
- `last_result_snapshot`
- `updated_at`

职责：

- 表示一个大纲项的一条长期会话
- 保存当前工作底稿与持久摘要

#### `outline_message`

字段建议：

- `id`
- `session_id`
- `role`
- `message_type`
- `content`
- `meta`
- `created_at`

职责：

- 记录每轮用户消息与系统回复

### 9.2 为什么不直接把历史塞进现有 synopsis / volume_plan 文档

原因有三点：

- 现有文档模型负责“最终结果”，不适合承载对话历史
- 会话摘要和消息历史是独立关注点
- 后续若继续优化上下文压缩，不应反复改动文档结构

因此应把“结果文档”和“对话会话”解耦。

### 9.3 结果写回

每次优化成功后：

- 更新当前会话的 `last_result_snapshot`
- 更新会话 `conversation_summary`
- 写入 `outline_message`
- 将新结果同步写回现有总纲或卷纲存储

这样既保留现有读取能力，也建立新的交互层。

## 10. 接口设计

建议新增 4 组接口。

### 10.1 大纲列表与详情

用途：

- 返回左侧列表
- 返回当前选中项详情

输出需覆盖：

- 统一 `outline items`
- 当前项状态
- 最近更新时间
- 结果快照

### 10.2 会话消息

用途：

- 按大纲项读取独立会话
- 返回最近若干条消息

第一版可以只取最近一页消息。

### 10.3 提交优化意见

输入：

- 当前大纲项标识
- 用户意见

行为：

- 根据大纲项类型选择 agent
- 组装上下文
- 执行一轮优化
- 返回更新后的详情快照和系统回复

第一版优先同步返回；若执行时间过长，再升级为异步任务。

### 10.4 会话摘要维护

会话摘要由后端在每轮成功后自动更新，不要求前端单独维护。

## 11. 前端改造

### 11.1 页面命名与路由

- 页面标题从 `总纲与卷规划` 改为 `大纲规划`
- 相关导航文案同步改名
- 视图文件可继续沿用 `VolumePlan.vue`，也可以在实现时重命名为更贴合的新名称

### 11.2 组件拆分建议

建议将页面拆为以下几个组件：

- `OutlineSidebar`
- `OutlineDetailPanel`
- `OutlineConversation`
- `OutlineEmptyState`

理由：

- 左侧列表逻辑独立
- 详情区需要兼容总纲与卷纲
- 会话区会持续增长，单独组件更易维护

### 11.3 默认交互

- 默认选中 `总纲`
- 点击左侧项时，右侧切换到对应结果和对应会话
- 优化中时保留旧结果，不先清空
- 成功后整体刷新当前项
- 失败时保留旧结果，并在会话区显示错误信息

---

## 12. 错误处理

### 12.1 无卷纲内容

- 不是错误
- 右侧显示明确空状态
- 仍允许用户输入“先生成当前卷卷纲”

### 12.2 本轮优化失败

- 保留旧结果
- 在会话区展示失败原因
- 支持用户再次提交

### 12.3 会话加载失败

- 只影响下半区
- 详情区继续可看
- 会话区支持重试

### 12.4 详情加载失败

- 只影响上半区
- 左侧与会话区继续可用
- 页面按区域降级，不整页白屏

---

## 13. 测试策略

### 13.1 前端测试

- 左侧列表正确显示总纲、已生成卷纲、未生成卷纲
- 切换 item 后，右侧加载对应详情和对应会话
- 提交意见后，执行中、成功、失败状态都正确呈现
- 未生成卷纲时，空状态和对话区仍可用

### 13.2 后端测试

- 能按大纲项维度创建和读取独立会话
- 提交总纲意见时调用 `BrainstormAgent`
- 提交卷纲意见时调用 `VolumePlannerAgent`
- 优化成功后能更新结果快照、消息记录、会话摘要
- 上下文组装不会把无关卷纲整段塞入 prompt

### 13.3 验收标准

用户打开 `大纲规划` 页面后，应能快速完成这些动作：

- 看清当前有哪些大纲项
- 切换到任意卷纲
- 直接用自然语言提出修改意见
- 在同一页面看到优化后的结果
- 切到另一卷时，不会混入上一卷的对话上下文

---

## 14. 实现边界与迭代顺序

建议本能力作为一个单独实现计划推进，顺序如下：

1. 页面命名与左右布局改造
2. 左侧统一列表与右侧详情切换
3. 会话模型与接口
4. 页面内提交优化意见
5. 上下文摘要与 prompt 压缩

这样可以先把页面骨架和数据模型稳定下来，再接入真正的对话优化能力。
